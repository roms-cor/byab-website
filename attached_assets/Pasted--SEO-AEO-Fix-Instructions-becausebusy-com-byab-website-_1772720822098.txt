# SEO/AEO Fix Instructions — becausebusy.com (byab-website)

> Ces tâches sont issues d'un audit SEO/AEO complet du codebase.  
> Elles sont triées par priorité décroissante. Chaque tâche est indépendante sauf mention contraire.

---

## 🔴 P0 — Bloquant pour l'indexation

---

### T001 · Swap route racine `/` → composant `Home`

- **Blocked By:** []
- **Details:**
  - Dans `App.tsx`, la route `path="/"` charge actuellement `ComingSoon`
  - Remplacer par le composant `Home` (actuellement sur `/home`)
  - Supprimer la route `/home` ou la rediriger vers `/` avec un `<Redirect>`
  - La page `ComingSoon` peut être conservée sous une route cachée type `/coming-soon` si besoin
  - **Files:** `client/src/App.tsx`
- **Acceptance:** Naviguer vers `https://becausebusy.com/` affiche le contenu de la Home page (hero, services, team, etc.), pas le placeholder "Coming soon"

---

### T002 · URLs absolues dans `sitemap.xml`

- **Blocked By:** []
- **Details:**
  - Les `<loc>` actuelles utilisent des chemins relatifs (`<loc>/</loc>`, `<loc>/home</loc>`)
  - Remplacer par des URLs absolues : `<loc>https://becausebusy.com/</loc>` et `<loc>https://becausebusy.com/home</loc>`
  - Après T001, mettre à jour le sitemap pour ne lister que `/` (plus de `/home` comme URL primaire)
  - Ajouter `<lastmod>` avec la date du dernier build au format `YYYY-MM-DD`
  - **Files:** `client/public/sitemap.xml`
- **Acceptance:** Toutes les `<loc>` commencent par `https://becausebusy.com`

---

### T003 · URL absolue dans `robots.txt`

- **Blocked By:** []
- **Details:**
  - La directive `Sitemap:` utilise un chemin relatif (`Sitemap: /sitemap.xml`)
  - Remplacer par : `Sitemap: https://becausebusy.com/sitemap.xml`
  - **Files:** `client/public/robots.txt`
- **Acceptance:** `robots.txt` contient `Sitemap: https://becausebusy.com/sitemap.xml`

---

### T004 · Supprimer `maximum-scale=1` du viewport meta

- **Blocked By:** []
- **Details:**
  - La balise viewport contient `maximum-scale=1` qui bloque le zoom sur mobile → pénalité Core Web Vitals accessibilité
  - Remplacer par : `<meta name="viewport" content="width=device-width, initial-scale=1.0">`
  - **Files:** `client/index.html`
- **Acceptance:** La balise viewport ne contient plus `maximum-scale`

### T005 · Ajouter `og:image` et `og:url` dans `index.html`

- **Blocked By:** []
- **Details:**
  - Ajouter dans le `<head>` statique de `index.html` :
    ```html
    <meta property="og:url" content="https://becausebusy.com/" />
    <meta property="og:image" content="https://becausebusy.com/og-image.png" />
    <meta property="og:image:width" content="1200" />
    <meta property="og:image:height" content="630" />

    ```
  - Créer un fichier image `og-image.png` (1200×630px) dans `client/public/` — peut être un screenshot du hero ou un visuel brandé simple avec le logo BYAB sur fond sombre
  - **Files:** `client/index.html`, `client/public/og-image.png`
- **Acceptance:** Un aperçu LinkedIn/Twitter du site affiche le visuel et non une carte texte vide

### T006 · Corriger le `BreadcrumbList` JSON-LD

- **Blocked By:** [T001]
- **Details:**
  - Le schéma `BreadcrumbList` actuel pointe vers `/home` et `/design` (page `noindex`)
  - Après T001, le corriger pour pointer vers les sections réelles de la homepage via ancres :
    ```json
    {
      "@type": "BreadcrumbList",
      "itemListElement": [
        { "@type": "ListItem", "position": 1, "name": "Accueil", "item": "https://becausebusy.com/" },
        { "@type": "ListItem", "position": 2, "name": "Services", "item": "https://becausebusy.com/#services" },
        { "@type": "ListItem", "position": 3, "name": "Références", "item": "https://becausebusy.com/#track-record" }
      ]
    }
    ```
  - Adapter les noms et ancres aux vraies sections présentes dans `home.tsx`
  - **Files:** `client/index.html`
- **Acceptance:** Le JSON-LD BreadcrumbList ne contient plus de références à `/design` ou `/home`

### T007 · Supprimer ou renseigner `telephone` dans le schema `ProfessionalService`

- **Blocked By:** []
- **Details:**
  - Le champ `"telephone": ""` est une chaîne vide dans le JSON-LD → erreur remontée dans Google Search Console
  - Si pas de numéro public : supprimer entièrement la propriété `telephone` du schéma
  - Si numéro disponible : renseigner au format international `"+33XXXXXXXXX"`
  - **Files:** `client/index.html`
- **Acceptance:** Le schema `ProfessionalService` ne contient plus `"telephone": ""`

---

### T008 · Corriger les URLs relatives dans les JSON-LD

- **Blocked By:** []
- **Details:**
  - Dans le schéma `Organization`, la propriété `"url": "/"` est une URL relative → invalide Schema.org
  - Remplacer toutes les URLs relatives par des URLs absolues dans tous les blocs JSON-LD :
    - `"url": "/"` → `"url": "https://becausebusy.com/"`
    - `"logo": "/logo.png"` → `"logo": "https://becausebusy.com/logo.png"`
    - Idem pour toute propriété `image`, `sameAs`, `url` dans les schémas `Organization`, `ProfessionalService`, `WebSite`
  - **Files:** `client/index.html`
- **Acceptance:** Aucune URL relative dans les blocs `<script type="application/ld+json">`

---

### T009 · Homogénéiser `areaServed` dans `ProfessionalService`

- **Blocked By:** []
- **Details:**
  - Le schéma `Organization` utilise des objets structurés pour `areaServed` mais `ProfessionalService` utilise des strings simples `["Paris", "La Rochelle", "France"]`
  - Mettre en cohérence avec le format structuré :
    ```json
    "areaServed": [
      { "@type": "City", "name": "Paris" },
      { "@type": "City", "name": "La Rochelle" },
      { "@type": "Country", "name": "France" }
    ]
    ```
  - **Files:** `client/index.html`
- **Acceptance:** Les deux schémas utilisent le même format d'objet pour `areaServed`
